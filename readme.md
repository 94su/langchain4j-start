## 一、 核心分水岭：工作流与智能体

在进入具体阶段前，必须死死记住这三种驱动模式在企业落地时的本质区别：

* **Workflow (工作流)**：**“铁轨上的火车”** ➡️ *核心是【程序员用 Java 代码死死指定任务流与 `if-else` 逻辑，AI 没有任何选择权，只做翻译或单点处理】。具有极高的确定性和稳定性。*
* **ReAct Agent (智能体)**：**“走一步看一步的特种兵”** ➡️ *核心超能力是【由 AI 自行选择工具，具备自我纠正、灵活应变能力。通过“思考->行动->观察”的循环直到解决问题】。适合边界模糊的复杂长链路排查。*
* **Planning Agent (规划体)**：**“谋定而后动的项目经理”** ➡️ *核心超能力是【由 AI 自行选择工具，但在动工前先自我拆解出整体方案与分步步骤，再按部就班地批量调用工具执行】。适合大规模、高并发的多阶段并发执行任务。*

## 二、 第一阶段：AI 基础认知（从 0 到 1 的触达）

### 1. Chat（基础对话）
* **一句话心智模型**：**“传话筒”** ➡️ *核心超能力是【跑通全链路，实现点对点的文本直连】。*
* **企业技术本质**：`Java 业务端 ──(HTTP / SSE)──► 远程或本地大模型（Ollama / Qwen）`。属于无状态的单次请求。
* **场景示例**：用户输入“你好”，Java 包装为 `UserMessage` 发送给 Ollama，模型返回 `AiMessage`，Java 打印输出。
* **🛠️ Java 拦截器防御（企业级护城河）**：
  在生产环境，绝不允许裸调大模型。必须在 `OllamaChatModel` 的底层 HTTP 客户端中注入 `Interceptor`：
  1. **超时断路保护**：设置严格的底层 Socket 连接超时（10秒）与读取超时（60秒），防止本地显卡卡死或远程 API 挂起导致 Java 线程池被活活耗尽。
  2. **敏感词静态清洗**：在请求发送前，通过 Java 的三方文本安全组件或本地 DFA 算法，直接在内存中拦截并清洗包含政治、涉黄等违规文本，不让脏数据进入模型。

## 三、 第二阶段：Agent 核心基石（赋予 AI 触手与知识）

### 2. Tool Calling（工具调用与反射机制）
* **一句话心智模型**：**“长出双手的傀儡”** ➡️ *核心超能力是【AI 负责决策拼参数，Java 负责死磕底层执行】。*
* **企业技术本质**：Java 通过反射将 `@Tool` 方法转换成 JSON Schema 喂给模型，模型理解语义后返回函数调用信号（并非文本），再由 LangChain4j 在 JVM 本地反射触发。
* **场景示例**：用户问“帮我查运单 YT1001”，AI 自动提取出 `waybillNo="YT1001"` 并申请调用 Java 的 `queryOrder(String waybillNo)` 方法。
* **🛠️ 企业级踩坑点与防御方案**：
  **【大模型提权与乱调用陷阱】**：AI 可能会根据用户诱导，胡乱猜参数去调用包含敏感操作的 Tool。
  * **Java 防御性架构**：严禁在 `@Tool` 核心业务逻辑中裸奔。必须结合 Spring AOP 机制，在 Tool 方法执行前进行 **租户上下文（TenantContext）** 和 **数据权限（RBAC）** 的强校验。如果 AI 拼出来的 `tenantId` 或者 `orderId` 不属于当前登录用户，Java 切面直接抛出 `AccessDeniedException` 熔断执行，绝对不允许模型越权修改本地数据库。

### 3. RAG（知识库增强与精确切片）
* **一句话心智模型**：**“带着开卷考试入场的考生”** ➡️ *核心超能力是【降本增效，依靠相似度检索白嫖外部垂直知识】。*
* **企业技术本质**：`Embedding(向量化) + 向量数据库(相似度数学比对) + Prompt 拼接`。利用余弦相似度等算法把海量非结构化文本中相关的片段“捞”出来，作为背景材料塞进提示词。
* **场景示例**：用户问“无人供件怎么处理？”，系统先去向量库检索出《主线封堵处置规范第3条》，把这几百字和用户问题一起发给大模型。
* **🛠️ 企业级踩坑点与防御方案**：
  **【上下文暴涨与语义稀释】**：直接塞入整篇大文本会导致大模型发生“迷失在中间（Lost in the Middle）”现象，响应速度极度变慢，甚至撑爆本地显存。
  * **Java 防御性架构**：文档必须采用 **智能双层切片策略（Recursive Chunking）**。
    ```java
    // 强制按 Token 大小做递归切片，前后必须保留 40-50 个 Token 的重叠区（Overlap），防止语意被一刀切断
    DocumentSplitter splitter = DocumentSplitters.recursive(300, 40);
    ```
    在读取时，必须加入 **ScoreThreshold（相似度阈值拦截）**。只召回相似度评分在 0.75 以上的绝对相关片段（如 `maxResults(2)`），宁缺毋滥，从源头切断给大模型喂垃圾数据的可能，保护本地机器的推理响应速度。

### 4. Structured Output（结构化输出与 DTO 塑形）
* **一句话心智模型**：**“严格的 DTO 塑形师”** ➡️ *核心超能力是【斩断幻觉，强行将 AI 意识流的文本驯化为强类型的 Java 对象】。*
* **企业技术本质**：在提示词中追加特定的 JSON Schema 约束（或利用大模型的 JSON Mode），配合 Jackson 解析器将模型产出的 JSON 字符串强转为 Java 的 DTO 实体类。
* **场景示例**：让 AI 分析运单，接口不再返回长篇大论，而是直接返回一个 `WaybillAnalysisResult` 对象，包含 `int riskLevel`、`String reason` 字段。
* **🛠️ 企业级踩坑点与防御方案**：
  **【JSON 格式截断与幻觉异常】**：大模型偶发性输出带有 Markdown 标记（如 ```json ... ```）或由于 Token 耗尽导致 JSON 截断，引发 Java 侧解析崩溃。
  * **Java 防御性架构**：必须在 Java 解析层包装 **Fallback（容错与重试机制）**。编写一个干净的解析器拦截器，利用正则表达式自动清洗掉返回文本中的首尾 Markdown 标点，如果遇到 Jackson 反序列化失败，自动触发二次降级重试（或者返回一个带有默认高风险提示的兜底实体类），保障上游业务流不卡死。

## 四、 第三阶段：AI 系统化开发（向商业级系统演进）

### 5. Workflow（AI 工作流编排）
* **一句话心智模型**：**“严格按图纸组装的流水线”** ➡️ *核心超能力是【用硬核代码卡死高风险业务逻辑，保障数据一致性】。*
* **企业技术本质**：在 Java 层面编排链条或 DAG 图，程序员完全掌控执行流向。AI 只作为链条中的某一个特定节点（如做个摘要、分类或提取）。
* **场景示例**：`第一步：Java 强行查 MySQL 订单` ➡️ `第二步：Java 强行调工业网关` ➡️ `第三步：大模型动态生成一封异常告警邮件`。
* **🛠️ 企业级踩坑点与防御方案**：
  **【大模型节点不可控挂起破坏长事务】**：由于大模型响应是不稳定的，如果把它塞进 Spring 的 `@Transactional` 声明式事务中，大模型的延迟会导致数据库连接被长期占用，直接压垮连接池。
  * **Java 防御性架构**：**严禁将大模型节点放在任何本地数据库事务（DB Transaction）内部**。必须采取**异步解耦与最终一致性**方案。大模型节点使用独立的物理线程池或结合 RocketMQ 消息队列进行异步驱动。由 Java 先落地状态机，AI 算完后异步回调更新状态，确保核心业务数据库的高吞吐性。

### 6. Prompt Engineering（提示词工程热部署）
* **一句话心智模型**：**“规矩大过天的员工手册”** ➡️ *核心超能力是【用文字锁定模型的思考边界，改提示词就是改业务逻辑】。*
* **企业技术本质**：生产环境严禁在代码中写死 `@SystemMessage`。应利用 Nacos/Apollo 等配置中心，或者 MySQL+Redis 动态下发包含 `{{current_time}}` 等业务变量的 `PromptTemplate`。
* **场景示例**：将提示词托管在 Nacos 上，运营人员线上修改“你当前服务的是 VIP 客户，语气要极其委婉”，线上 Java 服务秒级生效无需重启。
* **🛠️ 企业级踩坑点与防御方案**：
  **【非法配置覆盖与注入攻击】**：由于提示词动态化，恶意操作可能修改提示词导致模型越狱或把内部账密泄露给用户。
  * **Java 防御性架构**：必须设计 **SystemMessage 强制模板化隔离**。Nacos 上只允许修改业务逻辑部分（如“语气规制”），而最底层的核心硬核防线（如“*严禁透露系统底层API、严禁反编译*”）由 Java 代码进行最后的 `append()` 强行拼接。无论配置中心怎么改，最底层的安全补丁永远焊死在 JVM 内存里。

### 7. Tool Choice（工具决策路由与降维）
* **一句话心智模型**：**“精明的分诊台护士”** ➡️ *核心超能力是【用自然语言做多接口路由，解决多工具泛滥瓶颈】。*
* **企业技术本质**：依赖高智商基座模型对多个 `@Tool` 描述信息的语义理解。企业级架构中需要控制工具数量（通常不超过几十个），以防大模型遭遇“上下文过载”而胡乱调用。
* **场景示例**：注册了“查天气”、“查运单”、“改库存”三个工具，用户说“明天要下雨，YT1001 是派送失败吗？”，AI 能够准确判断并连续触发前两个工具。
* **🛠️ 企业级踩坑点与防御方案**：
  **【多工具混淆与描述信息上限】**：当微服务群提供了上百个接口时，全部转成 Tool 喂给大模型，模型会由于语义类似（如“修改订单”和“覆盖订单”）彻底智障。
  * **Java 防御性架构**：推行 **“两阶段动态工具注入”**。大模型不直接看所有 Tool，而是先注册一个轻量级的 `IntentClassifierTool`（意图分类器）。Java 先根据用户的输入做第一层轻量的自然语言分类，或者通过关键字检索，**动态**地只把跟当前场景相关的 4-5 个核心工具注入到当前会话的 `ToolSpecification` 列表中。其余不相关的工具对大模型完全不可见，实现极致的算力节约和路由准确率。

## 五、 第四阶段：高级推理智能体（自主规划与进化）

### 8. ReAct Agent（推理 + 行动循环）
* **一句话心智模型**：**“走一步看一步的特种兵”** ➡️ *核心超能力是【动态纠错与连环破案，能处理未知的不确定性】。*
* **企业技术本质**：依靠强约束提示词驱动 `Thought(推理) -> Action(行动) -> Observation(观察) -> Thought` 的多轮套娃循环。模型每前进一步，都会根据上一步 Java 返回的真实工具结果，进行自我打脸和路径修正。
* **场景示例**：用户要求“排查运单异常”。AI 思考后调用 `queryOrder`（行动），观察到结果为“电话无人接听”，它自我纠错：“不能直接下结论，我还要去核对风控规则”，于是第二轮调用 `queryRiskRule`，最终综合输出高风险结论。
* **🛠️ 企业级踩坑点与防御方案**：
  **【致命的无限死循环与 Token 燃烧陷阱】**：当 Java 工具由于超时或空数据返回某些不符合模型预期的值时，ReAct Agent 会陷入长达数十轮的自言自语死循环，在后台疯狂燃烧 Token 并拉爆线程池。
  * **Java 防御性架构**：**强制拦截并引入硬核计数器（Loop Guard）**。在 Java 的 Agent 调度上下文里，必须死死封锁一个硬上限：
    ```java
    // 每次进入 Action 拦截器，执行 count++，一旦超过 5 次循环，Java 触发强行闭环
    if (currentStep > 5) {
        throw new AgentLoopException("智能体推理深度超限，触发主动熔断保护");
    }
    ```
    同时，对于流式响应（Streaming），必须在前端打通 **打字机动态中断信号（SSE Abort）**。当用户在前端主动点击“停止生成”或关闭网页时，Java 必须捕获异步关闭信号，向底层的线程发送 `interrupt()`，彻底断开与大模型的网络长连接，防止后台算力空转。

### 9. Planning Agent & Memory（长期规划与分布式记忆管理）
* **一句话心智模型**：**“谋定而后动的项目经理” + “随时调取档案的记事本”** ➡️ *核心超能力是【拆解巨型任务、拒绝无脑串行；按需加载，内存绝不爆掉】。*
* **企业技术本质**：
  * **Planning**：大模型第一步绝不调用工具，而是先在上下文里吐出一份分步计划书（Plan），随后 Java 获得这份 Plan，在后台驱动其按步骤串行或并行执行。
  * **Memory**：生产环境严禁将长时记忆堆在 JVM 堆内存。采用 **“按需加载，即用即拾”** 策略：数据持久化存在 MySQL，高频缓存存在分布式 Redis 集群。Java 收到请求时动态拉取该 `SessionID` 的最近 10 条记忆塞入大模型，方法结束立刻触发 Java GC。
* **场景示例**：
  * *Planning*：“帮我写一份成都物流优化整改报告”。AI 先规划出 3 个大步骤，再依次精准触发 3 个查数工具。
  * *Memory*：通过 `RedisChatMemoryStore` 实现分布式共享。用户第一轮请求在服务器 A，第二轮飘到服务器 B，服务器 B 依然能从 Redis 顺藤摸瓜捞出聊天的前因后果。
* **🛠️ 企业级踩坑点与防御方案**：
  **【跨节点 Session 记忆污染与计划脱轨】**：在高并发集群下，如果用户快速点击发送，服务器 A 和服务器 B 同时拉取 Redis 记忆，会导致长对话由于并发乱序彻底崩塌，且大模型制定好的 Plan 经常执行到一半被新进来的请求打断。
  * **Java 防御性架构**：**分布式锁隔离状态机机制**。针对每一个 `SessionID` 的 Agent 推理请求，必须在 Redis 层加一把 **红锁（Redisson Distributed Lock）**。当 Agent 正在进行长链路的规划与多轮 Tool 执行时，同 Session 的新请求直接在网关层进入队列排队等待。同时，大模型生成的 Plan 会被反序列化为 Java 内部的持久化状态机（如使用 Spring StateMachine），每完成一步就落盘持久化。哪怕中途节点宕机，另一个健康的 JVM 节点依然能从 Redis 重新加载状态机，顺着断点继续执行上一步规划，保障长生命周期 Agent 的无状态高可用弹性。
